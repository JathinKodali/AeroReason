1. PROJECT OVERVIEW
================================================================================

Project Name  : AeroReason (MINOR)
Purpose       : End-to-end predictive maintenance system for aircraft turbofan
                engines using the NASA C-MAPSS (Commercial Modular Aero-
                Propulsion System Simulation) dataset.
Architecture  : Full-stack web application with a Python/FastAPI backend and a
                React/Vite frontend.
AI Approach   : Hybrid — combines deep learning (LSTM) for numerical RUL
                (Remaining Useful Life) prediction with a rule-based reasoning
                engine AND an optional LLM (Ollama/llama3) for context-aware,
                human-readable maintenance insights.

Key Value Proposition:
  - Predicts how many operational cycles an engine has left before failure.
  - Classifies engines into risk categories (LOW / MODERATE / CRITICAL).
  - Generates actionable maintenance recommendations.
  - Provides an interactive, modern web dashboard for fleet-wide and per-engine
    health monitoring.
  - Supports uploading custom CSV sensor data for real-time inference.

3. DATASET — NASA C-MAPSS
================================================================================

Source: NASA Prognostics Center of Excellence
URL: https://data.nasa.gov/dataset/C-MAPSS-Aircraft-Engine-Simulator-Data

The dataset simulates turbofan engine degradation from healthy to failure. It
contains four sub-datasets with varying operational conditions and fault modes:

  Sub-dataset | Train Engines | Test Engines | Op Conditions | Fault Modes
  ------------|---------------|--------------|---------------|------------
  FD001       |  100          |  100         | 1             | 1 (HPC)
  FD002       |  260          |  259         | 6             | 1 (HPC)
  FD003       |  100          |  100         | 1             | 2 (HPC+Fan)
  FD004       |  249          |  248         | 6             | 2 (HPC+Fan)

Files per sub-dataset:
  - train_FDxxx.txt  — Full run-to-failure sensor time series
  - test_FDxxx.txt   — Censored time series (cut off before failure)
  - RUL_FDxxx.txt    — Ground truth RUL values for the last cycle of each
                       test engine

Each row has 26 columns:
  - engine_id         : Unit number
  - cycle             : Time in cycles
  - op_setting_1..3   : Operational settings (altitude, Mach, throttle)
  - sensor_1..21      : 21 sensor measurements (temperatures, pressures,
                        speeds, flow rates, etc.)

The data is stored in the /Research/ directory along with:
  - "Damage Propagation Modeling.pdf" — Research paper on the simulation
  - hybrid_predictive_maintenance_research_notebook.ipynb — Jupyter notebook
    with exploratory research

5.1 Data Preprocessing (preprocessing/data_loader.py)
----------------------------------------------------------------------
This module handles all data ingestion and transformation:

Column Definitions:
  - INDEX_COLS    : ["engine_id", "cycle"]
  - SETTING_COLS  : ["op_setting_1", "op_setting_2", "op_setting_3"]
  - SENSOR_COLS   : ["sensor_1" through "sensor_21"]
  - ALL_COLS      : All 26 columns combined

Dropped Features (near-constant, no degradation signal):
  - Sensors: sensor_1, sensor_5, sensor_6, sensor_10, sensor_16, sensor_18,
             sensor_19
  - Settings: op_setting_3

Remaining Features (FEATURES): 16 columns used for model input.

RUL Cap: 125 cycles (standard C-MAPSS practice — piecewise linear labeling
  where RUL is capped at 125 so the model doesn't need to learn the flat
  "healthy" portion of the curve).

Key Functions:
  - load_data(data_dir, dataset_id)
      Loads train_FDxxx.txt, test_FDxxx.txt, RUL_FDxxx.txt as DataFrames.

  - add_rul_labels(df, cap=125)
      Computes RUL for training data: RUL = max_cycle - current_cycle, clipped
      to [0, 125].

  - add_test_rul(test_df, rul_df, cap=125)
      Attaches ground-truth RUL to test data using the RUL file.

  - normalize_features(train_df, test_df, features=FEATURES)
      Fits a MinMaxScaler on training data, transforms both train and test.
      Returns the scaler for later reuse.

  - save_scaler(scaler, path) / load_scaler(path)
      Persists/loads the fitted scaler using joblib.

  - normalize_with_scaler(df, scaler, features)
      Applies an already-fitted scaler to new data (for user uploads).

  - create_sequences(df, window=30, features=FEATURES)
      Creates sliding-window sequences for LSTM training. For each engine,
      slides a window of 'window' cycles and uses the last row's RUL as label.
      Output shape: (num_samples, window, num_features)

  - create_test_sequences(df, window=30, features=FEATURES)
      Creates ONE sequence per engine from the LAST 'window' cycles. If an
      engine has fewer cycles, it is zero-padded at the front.
      Output shape: (num_engines, window, num_features)

  - prepare_data(data_dir, dataset_id, window=30)
      Full pipeline convenience wrapper: load → label → normalize → sequence.
      RUL labels are normalized to [0, 1] by dividing by RUL_CAP.
      Returns: X_train, y_train, X_test, y_test, test_engine_ids, scaler,
               train_df, test_df


5.2 LSTM Model (models/lstm_model.py)
----------------------------------------------------------------------
Architecture (Keras Sequential):
  Layer 1: LSTM(128 units, return_sequences=True)    → learns temporal patterns
  Layer 2: Dropout(0.2)                               → regularization
  Layer 3: LSTM(64 units, return_sequences=False)     → sequence compression
  Layer 4: Dropout(0.2)                               → regularization
  Layer 5: Dense(32, activation='relu')               → hidden representation
  Layer 6: Dense(1, activation='sigmoid')             → RUL output in [0, 1]

Training Configuration:
  - Optimizer: Adam (learning_rate=0.001)
  - Loss: Mean Squared Error (MSE)
  - Epochs: 50 (default, configurable)
  - Batch Size: 256
  - Validation Split: 20%
  - Early Stopping: patience=10, restore_best_weights=True

Key Functions:
  - build_lstm_model(input_shape, lstm_units_1=128, lstm_units_2=64,
                     dropout_rate=0.2, learning_rate=0.001)
      Builds and compiles the model.

  - train_model(model, X_train, y_train, epochs=50, batch_size=256,
                validation_split=0.2, patience=10)
      Trains with early stopping. Returns Keras History object.

  - evaluate_model(model, X_test, y_test)
      Returns normalized RMSE and predictions.

  - save_trained_model(model, path) / load_trained_model(path)
      Save/load the Keras model to/from .keras files.

Per-Dataset Models:
  Separate models are trained for each sub-dataset (FD001–FD004) and stored as
  trained_model_FD001.keras through trained_model_FD004.keras. The API server
  automatically selects the per-dataset model if it exists, otherwise falls back
  to the generic trained_model.keras.


5.3 Reasoning Engine (reasoning/reasoning_engine.py)
----------------------------------------------------------------------
Converts numerical RUL predictions into actionable maintenance insights.

Risk Classification Thresholds:
  - CRITICAL : RUL ≤ 50 cycles    (color: #e74c3c — red)
  - MODERATE : RUL ≤ 100 cycles   (color: #f39c12 — amber)
  - LOW      : RUL > 100 cycles   (color: #2ecc71 — green)

MaintenanceInsight Data Class:
  Fields: engine_id, predicted_rul, risk_level, risk_color, explanation,
          recommendation, trend_summary (optional)

Rule-Based Templates:
  - LOW: "Engine is operating within normal parameters..." / Continue routine
    monitoring, next inspection in 50-100 cycles.
  - MODERATE: "Engine shows signs of degradation..." / Schedule inspection in
    20-30 cycles, increase monitoring, pre-order parts.
  - CRITICAL: "Engine is approaching failure..." / Ground engine, initiate
    replacement, notify crew, do NOT operate beyond 10 cycles.

Trend Analysis:
  - analyse_degradation_trend(engine_df, sensor_cols)
  - Uses np.polyfit (linear regression slope) on each sensor time series.
  - Sensors with |slope| > 0.001 are flagged as degrading.
  - Returns human-readable summary of degrading vs stable sensors.

LLM Enhancement:
  - When use_llm=True AND Ollama is available, the engine generates richer
    explanations, recommendations, and trend analyses using the llama3 model.
  - Falls back gracefully to rule-based templates if LLM is unavailable.

Key Functions:
  - classify_risk(rul) → (risk_level, color)
  - generate_explanation(rul, risk_level) → string
  - generate_recommendation(rul, risk_level) → string
  - reason(engine_id, predicted_rul, engine_df, sensor_cols, use_llm, llm_model)
      Full reasoning pipeline for one engine. Returns MaintenanceInsight.
  - batch_reason(engine_ids, predictions, test_df, sensor_cols, use_llm, llm_model)
      Runs reasoning for multiple engines.


5.4 Ollama LLM Engine (reasoning/ollama_engine.py)
----------------------------------------------------------------------
Integrates a local Ollama LLM (default: llama3) for generating rich, context-
aware maintenance insights.

Configuration:
  - OLLAMA_BASE_URL: http://localhost:11434
  - DEFAULT_MODEL: "llama3"

System Prompt:
  "You are an expert aerospace maintenance engineer AI assistant. You analyse
   turbofan engine health data and provide clear, actionable insights to
   maintenance crews. Be concise but thorough. Use professional language
   appropriate for aviation maintenance teams."

LLM Call Parameters:
  - Temperature: 0.4 (low creativity for factual, professional output)
  - Max Tokens (num_predict): 256
  - Stream: false (waits for complete response)
  - Timeout: 60 seconds

Health Check:
  - is_ollama_available() — checks GET /api/tags
  - get_available_models() — lists installed models

Prompt Types:
  1. Explanation Prompt: 2-3 sentence engine health status summary.
  2. Recommendation Prompt: 3-5 bullet-point maintenance actions.
  3. Trend Prompt: 2-3 sentence expert analysis of sensor degradation
     (compressor wear, turbine erosion, bearing fatigue, etc.).

Key Functions:
  - generate_llm_explanation(engine_id, rul, risk_level, trend_summary, model)
  - generate_llm_recommendation(engine_id, rul, risk_level, trend_summary, model)
  - generate_llm_trend_analysis(engine_id, rul, risk_level, trend_summary, model)
  All return Optional[str] — None on failure (caller falls back to rule-based).

5.6 FastAPI Server (api_server.py)
----------------------------------------------------------------------
Title: "AeroReason API", Version 1.0.0
CORS: All origins allowed (development mode).
Host Binding: 0.0.0.0 (accessible from all network interfaces)
Log Level: info (set via uvicorn startup arguments in run.sh)

Internal Caching:
  Dataset and prediction results are cached in-memory (_cache dict) to avoid
  redundant computation between requests.

API Endpoints:

  GET /api/datasets
    Returns: {"datasets": ["FD001", "FD002", "FD003", "FD004"]}

  GET /api/model/status
    Returns: {"model_exists": true/false}
    Checks if any per-dataset trained model file exists.

  GET /api/ollama/status
    Returns: {"online": true/false, "models": ["llama3", ...]}
    Checks Ollama server reachability and lists available models.

  GET /api/predictions?dataset=FD001&window=30
    Returns fleet-wide predictions:
    {
      "rmse": float,
      "total_engines": int,
      "critical": int, "moderate": int, "low": int,
      "fleet": [
        {"engine_id": int, "predicted_rul": float, "actual_rul": float,
         "error": float, "risk_level": string},
        ...
      ]
    }

  GET /api/engine/{engine_id}?dataset=FD001&window=30&use_llm=false&llm_model=llama3
    Returns detailed single-engine analysis:
    {
      "engine_id": int,
      "predicted_rul": float, "actual_rul": float,
      "risk_level": string, "risk_color": string,
      "explanation": string, "recommendation": string,
      "trend_summary": string,
      "sensors": [list of sensor names],
      "sensor_data": {
        "sensor_X": {"cycles": [..], "values": [..]},
        ...
      }
    }

  GET /api/performance?dataset=FD001&window=30
    Returns model performance metrics:
    {
      "rmse": float, "mean_error": float, "std_error": float,
      "engines_tested": int,
      "scatter": {"actual": [...], "predicted": [...]},
      "histogram": {"counts": [...], "bins": [...]}

  POST /api/predict/upload?window=30&use_llm=false&llm_model=llama3
    Accepts CSV file upload (multipart form data).
    Required CSV columns: engine_id, cycle, + at least 5 sensor columns.
    Uses FD001 model as the default/fallback model for uploaded data.
    Returns:
    {
      "total_engines": int, "avg_rul": float,
      "critical": int, "moderate": int, "low": int,
      "features_detected": int, "rows_loaded": int,
      "results": [
        {"engine_id": int, "predicted_rul": float, "risk_level": string,
         "explanation": string, "recommendation": string,

5.7 CLI Pipeline (main.py)
----------------------------------------------------------------------
Usage:
  python main.py                   # Train on FD001 and evaluate
  python main.py --dataset FD002   # Use a different sub-dataset
  python main.py --epochs 100      # Override training epochs
  python main.py --load            # Load pre-trained model instead

Arguments:
  --data-dir    : Path to data directory (default: ./Research)
  --dataset     : FD001 | FD002 | FD003 | FD004 (default: FD001)
  --window      : Sliding window size (default: 30)
  --epochs      : Max training epochs (default: 50)
  --batch-size  : Training batch size (default: 256)
  --model-path  : Path to save/load model (default: data/trained_model.keras)
  --load        : Flag to load existing model

Pipeline Steps:
  1. Data Preparation: load → RUL labeling → normalization → windowing
  2. Model Training (or loading): build LSTM → fit → save
  3. Evaluation: predict → denormalize → compute RMSE → save plots
  4. Reasoning: run batch reasoning on all test engines → print risk summary

Note: The CLI pipeline's completion message references a legacy Streamlit
  app path. The primary user interface is now the React/Vite frontend
  accessed via the web dashboard (http://localhost:5173).

12. KEY ALGORITHMS AND APPROACHES
================================================================================

12.1 Piecewise Linear RUL Labeling:
  - For training data: RUL = max_cycle_of_engine - current_cycle
  - Capped at 125 to avoid the model spending capacity learning the flat
    "healthy" region where RUL would be very high.
  - This creates a bent curve: flat at 125, then linearly decreasing to 0.

12.2 Feature Selection:
  - 7 sensors dropped (near-constant values, no degradation signal):
    sensor_1, 5, 6, 10, 16, 18, 19
  - 1 setting dropped: op_setting_3
  - Remaining 16 features used for model input.

12.3 Sliding Window Sequences:
  - Window size = 30 cycles (configurable).
  - For training: overlapping windows across all cycles of each engine.
  - For testing: single window from the last 30 cycles of each engine.
  - Zero-padding for engines with fewer cycles than the window size.

12.4 Normalization:
  - MinMaxScaler fit on training data, applied to both train and test.
  - RUL labels also normalized to [0, 1] by dividing by RUL_CAP (125).
  - Predictions are denormalized by multiplying by 125.

12.5 Risk Classification:
  - CRITICAL: predicted RUL ≤ 50 cycles
  - MODERATE: 50 < predicted RUL ≤ 100 cycles
  - LOW: predicted RUL > 100 cycles

12.6 Trend Analysis:
  - Linear regression (np.polyfit degree 1) on each sensor's time series.
  - Slope threshold: |slope| > 0.001 indicates degradation.
  - Minimum 10 data points required for analysis.

12.7 Hybrid AI Pipeline:
  Rule-Based (default):
    → Predefined templates for each risk level.
    → Fast, deterministic, always available.

  LLM-Enhanced (when Ollama is active):
    → Sends engine context (RUL, risk, trends) to llama3.
    → Gets professional aerospace maintenance language.
    → Falls back to rule-based if LLM call fails.
    → Temperature=0.4 for factual, consistent output.

15. NOTABLE DESIGN DECISIONS
================================================================================

  1. Per-Dataset Models: Separate LSTM models for each C-MAPSS sub-dataset
     (FD001-FD004) because they have different operational conditions and fault
     modes. The API automatically selects the correct model.

  2. Hybrid Rule-Based + LLM: The rule-based engine ensures the system always
     works without external dependencies. The LLM (Ollama) adds richer,
     more natural explanations when available.

  3. AI Reasoning Enabled by Default: useLlm defaults to true in the frontend,
     providing the best experience when Ollama is running.

  4. In-Memory Caching: The API server caches dataset loading and prediction
     results to avoid re-computation on repeated requests.

  5. Frontend Design: Dark-mode cinematic aesthetic with glassmorphism,
     animated orbs, and smooth page transitions — designed to feel premium
     and modern for presentation/review settings.

  6. CSV Upload Support: Users can upload their own engine sensor data (in
     C-MAPSS format) for inference, making the system useful beyond the
     built-in datasets.

  7. Graceful LLM Fallback: All LLM-powered functions return Optional[str]
     (None on failure). The reasoning engine transparently falls back to
     rule-based templates without user-visible errors.

  8. SEO Metadata: The frontend index.html includes proper meta description
     and title tags for search engine discoverability.

  9. Network Accessibility: The uvicorn server binds to 0.0.0.0 allowing
     access from other devices on the same network, useful for demos.

  10. Upload Model Fallback: Custom CSV uploads always use the FD001 model
      as a default, ensuring inference works regardless of which sub-dataset
      the user's data resembles.
